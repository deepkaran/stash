Failover mechanics

GsiClient retry mechanism for scan failures:

1. GsiClient within the query process maintains a connection pool of TCP connections to each index service node.
2. Each index scan selects an index replica and grabs a connection from the pool for execution. 
3. Scan request is sent to the Indexer. 
4. If there is any error returned (TCP error or from indexer), and no result entry has been returned to query, GsiClient will retry the scan on a different replica if available.
5. If any result entry has been returned from the GsiClient to query, no retry will happen.

Index node failover:
1. GsiClient subscribes to streaming notifications from local ns-server REST endpoint.
2. Any topology change or node status change will be detected as a notification.
3. GsiClient will find the latest list of all indexers (excluding "unhealthy"). Any failed over indexer will get excluded here.
4. GsiClient will update its metadata cache to exclude the index metadata for the failed over/unhealthy nodes.
5. Also, for any failed over or unhealthy nodes, the connection pool gets discarded and connections are closed. Any ongoing/waiting scans for those nodes will get aborted and return error. These scans are eligible for retry if no result has been returned.


Notes on node.Status:
1. If node is not reachable - Unhealthy
2. If memcached is down - Unhealthy
3. If index service is down - Healthy
